# Обзор проекта

Этот проект — **Kotlin Multiplatform (KMP)** приложение с отдельными оболочками для **Android, iOS и Web**, а также общей частью, содержащей бизнес-логику, работу с данными и инфраструктуру фич.

---

## 🧩 Общая архитектура

Проект построен по модульному принципу с разделением ответственности:

* **Домен** → бизнес-логика и контракты
* **Данные** → работа с API и базой данных
* **Ядро** → сборка приложения и кроссплатформенная логика
* **Платформенные приложения** → UI и платформенные особенности

---

## 📦 Структура модулей

| Модуль             | Описание                             | Когда изменять                              |
| ------------------ | ------------------------------------ | ------------------------------------------- |
| `android/app`      | Android-приложение (Jetpack Compose) | Android-интерфейс, manifest, запуск приложения |
| `ios/app`          | iOS-приложение (SwiftUI)             | iOS-интерфейс и нативная интеграция         |
| `web-app`          | Web-приложение (Angular + Nx)        | Web-интерфейс и фронтенд-инфраструктура     |
| `kmp/app`          | Общий фасад приложения               | Старт общего приложения и root factory      |
| `kmp/navigation`   | Общий фасад навигации                | Контракты корневой навигации                |
| `kmp/bridge-web`   | Модуль web-моста                     | Экспорт JS bridge + вывод KSP               |
| `kmp/bridge-apple` | Модуль Apple-моста                   | Ориентированный на iOS API общей фабрики    |
| `kmp/core`         | Устаревший слой композиции           | Совместимость в период декомпозиции         |
| `kmp/data`         | Слой данных                          | API, база данных, реализация репозиториев   |
| `kmp/domain`       | Доменный слой                        | Модели, use-case'ы, интерфейсы репозиториев |
| `kmp/feature/base` | Базовые примитивы фич                | FlowMVI-рантайм, плагины, lifecycle-хелперы |

## 🔗 Зависимости модулей

```text
android/app -> kmp/app
android/app -> kmp/navigation
kmp/app -> kmp/core
kmp/navigation -> kmp/core
kmp/bridge-web -> kmp/core
kmp/bridge-apple -> kmp/core
kmp/core -> kmp/data
kmp/core -> kmp/domain
kmp/core -> kmp/feature/base
kmp/data -> kmp/domain
web-app -> kmp/bridge-web (JS сборка)
ios/app -> Shared framework bridge
```

---

## 🧭 Правила работы

Следуйте этим принципам при разработке:

* Выбирайте **минимальный модуль**, в котором можно решить задачу
* Не трогайте общие модули, если изменение касается только UI конкретной платформы
* Логика распределяется так:

    * **Бизнес-логика → `kmp/domain`**
    * **Данные (API/БД) → `kmp/data`**
    * **Состояние, навигация, композиция → `kmp/core`**

---

## ⚠️ Важные ограничения

* `android/app` **не должен напрямую зависеть** от `kmp/data` или `kmp/domain` — только через `kmp/core`
* `web-app` и `ios/app` должны быть **тонкими слоями**:

    * без дублирования бизнес-логики
    * только адаптация UI к общей логике

---

## 🧱 Особенности платформ

* `kmp/core` экспортирует общий bridge:

    * для Web (JS)
    * для iOS (framework)
* `kmp/data` содержит платформенные реализации:

    * HTTP-клиенты (Ktor)
    * драйверы базы данных (SQLDelight)
* `kmp/domain` и `kmp/feature/base` — полностью общие модули (с подходом common-first)

---

## 🛠 Перед внесением изменений

Перед тем как что-то менять:

1. Ознакомьтесь с `build.gradle.kts` нужного модуля
2. Проверьте локальные README (особенно в `web-app` и `ios/app`)
3. Убедитесь, что изменения вносятся в **правильный слой архитектуры**

> ⚠️ Не используйте `.ai/mcp/mcp.json` как документацию проекта — это не источник архитектуры.
