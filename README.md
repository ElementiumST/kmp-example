# Project Overview

Этот проект — **Kotlin Multiplatform (KMP)** приложение с отдельными оболочками для **Android, iOS и Web**, а также общей частью, содержащей бизнес-логику, работу с данными и инфраструктуру фич.

---

## 🧩 Общая архитектура

Проект построен по модульному принципу с разделением ответственности:

* **Domain** → бизнес-логика и контракты
* **Data** → работа с API и базой данных
* **Core** → сборка приложения и кроссплатформенная логика
* **Platform apps** → UI и платформенные особенности

---

## 📦 Структура модулей

| Модуль             | Описание                             | Когда изменять                              |
| ------------------ | ------------------------------------ | ------------------------------------------- |
| `android/app`      | Android-приложение (Jetpack Compose) | Android UI, manifest, запуск приложения     |
| `ios/app`          | iOS-приложение (SwiftUI)             | iOS UI и нативная интеграция                |
| `web-app`          | Web-приложение (Angular + Nx)        | Web UI и фронтенд-инфраструктура            |
| `kmp/core`         | Общий слой приложения                | Навигация, композиция, связывание слоев     |
| `kmp/data`         | Слой данных                          | API, база данных, реализация репозиториев   |
| `kmp/domain`       | Доменный слой                        | Модели, use-case'ы, интерфейсы репозиториев |
| `kmp/feature/base` | Базовые примитивы фич                | MVI, состояние, переиспользуемая логика     |

---

## 🔗 Зависимости модулей

```text
android/app -> kmp/core
kmp/core -> kmp/data
kmp/core -> kmp/domain
kmp/core -> kmp/feature/base
kmp/data -> kmp/domain
web-app -> kmp/core (JS сборка)
ios/app -> SharedCore.framework (из kmp/core)
```

---

## 🧭 Правила работы

Следуйте этим принципам при разработке:

* Выбирайте **минимальный модуль**, в котором можно решить задачу
* Не трогайте shared-модули, если изменение касается только UI конкретной платформы
* Логика распределяется так:

    * **Бизнес-логика → `kmp/domain`**
    * **Данные (API/БД) → `kmp/data`**
    * **Состояние, навигация, композиция → `kmp/core`**

---

## ⚠️ Важные ограничения

* `android/app` **не должен напрямую зависеть** от `kmp/data` или `kmp/domain` — только через `kmp/core`
* `web-app` и `ios/app` должны быть **тонкими слоями**:

    * без дублирования бизнес-логики
    * только адаптация UI к общей логике

---

## 🧱 Особенности платформ

* `kmp/core` экспортирует общий bridge:

    * для Web (JS)
    * для iOS (framework)
* `kmp/data` содержит платформенные реализации:

    * HTTP-клиенты (Ktor)
    * драйверы базы данных (SQLDelight)
* `kmp/domain` и `kmp/feature/base` — полностью общие модули (common-first)

---

## 🛠 Перед внесением изменений

Перед тем как что-то менять:

1. Ознакомьтесь с `build.gradle.kts` нужного модуля
2. Проверьте локальные README (особенно в `web-app` и `ios/app`)
3. Убедитесь, что изменения вносятся в **правильный слой архитектуры**

> ⚠️ Не используйте `.ai/mcp/mcp.json` как документацию проекта — это не источник архитектуры.
