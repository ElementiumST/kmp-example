# Как работают корутины на разных языках

## Общая идея

Корутина - это функция, которую можно приостановить и позже продолжить с того же места. В отличие от обычной функции, она не обязана выполниться целиком за один проход.

Обычно корутины используют для:

- асинхронного I/O;
- конкурентных задач;
- генерации значений по мере необходимости;
- UI- и серверной логики без блокировки потока.

Во многих языках корутины позволяют писать асинхронный код в последовательном стиле, без ручной работы с колбэками.

## Что происходит внутри

Во многих современных языках корутина компилируется в машину состояний:

1. функция начинает выполняться как обычно;
2. доходит до точки приостановки (`await`, `suspend`, `co_await`);
3. сохраняет текущее состояние;
4. отдает управление runtime или event loop;
5. позже продолжает выполнение с сохраненного места.

Ключевая идея в том, что корутина сама по себе обычно не равна отдельному потоку. Часто тысячи корутин могут выполняться поверх небольшого числа реальных потоков.

## Kotlin

В Kotlin корутины строятся вокруг `suspend`, `CoroutineScope` и `Dispatcher`.

```kotlin
suspend fun loadUser(): User {
    val data = api.fetch()
    return map(data)
}
```

Как это работает:

- `suspend`-функция компилируется в машину состояний;
- при вызове `delay` или другого suspend API поток не блокируется;
- продолжение хранится в `Continuation`;
- выполнение возобновляется через `Dispatcher`, например `Main`, `IO` или `Default`.

Особенности:

- корутина не равна потоку;
- переключение происходит только в точках приостановки;
- удобно организованы structured concurrency и отмена задач.

## Python

В Python используется модель `async def` и `await`, обычно вместе с `asyncio`.

```python
async def load_user():
    data = await api.fetch()
    return data
```

Как это работает:

- `async def` создает coroutine object;
- выполнение идет через event loop;
- `await` временно отдает управление циклу событий;
- когда результат готов, event loop продолжает корутину.

Особенности:

- это кооперативная многозадачность;
- если внутри `async`-функции запустить тяжелый CPU-код без `await`, можно заблокировать loop;
- для CPU-bound задач обычно используют потоки, процессы или executors.

## JavaScript / TypeScript

В JavaScript чаще говорят не про корутины напрямую, а про `async/await`, но по сути модель очень близкая.

```ts
async function loadUser() {
  const data = await fetch("/api/user");
  return data.json();
}
```

Как это работает:

- `async`-функция возвращает `Promise`;
- `await` приостанавливает выполнение до завершения promise;
- продолжение ставится в очередь event loop, обычно в microtask queue.

Особенности:

- `await` не создает новый поток;
- асинхронность обеспечивается браузером или Node.js runtime;
- модель удобна для I/O, но не для тяжелых вычислений в одном потоке.

## C#

В C# модель `async/await` тесно связана с `Task`.

```csharp
async Task<User> LoadUserAsync() {
    var data = await api.FetchAsync();
    return data;
}
```

Как это работает:

- компилятор тоже превращает метод в машину состояний;
- `await` подписывается на завершение `Task`;
- после завершения выполнение продолжается с нужного места.

Особенности:

- continuation может вернуться в UI-поток через `SynchronizationContext`;
- поведение продолжения можно менять через `ConfigureAwait(false)`;
- async-модель глубоко встроена в .NET экосистему.

## Swift

В `Swift` нет слова "корутины" в повседневной терминологии, но `async/await` и `Task` решают очень похожую задачу.

```swift
func loadUser() async throws -> User {
    let data = try await api.fetch()
    return map(data)
}
```

Как это работает:

- `async`-функция может быть приостановлена в точке `await`;
- компилятор и runtime сохраняют нужное состояние функции;
- выполнение продолжается позже, когда асинхронная операция завершится;
- запуск параллельной работы обычно идет через `Task`, `async let` или `TaskGroup`.

Особенности:

- модель близка к `Kotlin` и `C#` по синтаксису;
- structured concurrency встроена в язык;
- для изоляции состояния используются `actor`;
- есть понятие `MainActor` для кода, который должен выполняться на главном потоке.

## C++20

В C++20 корутины встроены в язык, но конкретную модель исполнения задает библиотека.

```cpp
task<int> load_user() {
    auto data = co_await fetch();
    co_return 42;
}
```

Как это работает:

- `co_await`, `co_yield` и `co_return` превращают функцию в coroutine;
- компилятор создает coroutine frame со всем нужным состоянием;
- дальнейшее поведение определяется типами `promise`, `awaiter` и используемой библиотекой.

Особенности:

- механизм очень гибкий;
- язык почти не навязывает конкретный runtime;
- подход мощный, но сложнее в практическом применении, чем в Kotlin или Python.

## Rust

В Rust `async` тоже компилируется в машину состояний, но работает через `Future`.

```rust
async fn load_user() -> User {
    let data = fetch().await;
    data
}
```

Как это работает:

- `async fn` возвращает объект, реализующий `Future`;
- пока runtime не вызывает `poll`, выполнение не идет;
- если результат не готов, future возвращает `Pending`;
- когда ресурс готов, runtime будит future через `Waker`.

Особенности:

- модель очень явная и производительная;
- обычно используется вместе с Tokio или async-std;
- async тесно связан с системой владения памятью и `Pin`.

## Go

В Go обычно говорят не о корутинах, а о goroutines. Это близкая, но не идентичная модель.

```go
go loadUser()
```

Как это работает:

- `go` запускает функцию конкурентно;
- goroutine дешевле системного потока;
- встроенный runtime Go сам планирует выполнение;
- взаимодействие между задачами часто идет через `channel`.

Главное отличие от `async/await`-модели:

- goroutine больше похожа на легковесный поток;
- приостановка и возобновление управляются runtime, а не только явными точками `await`;
- разработчик реже думает в терминах continuation.

## iOS: что важно на практике

На `iOS` чаще говорят не столько о "корутинах", сколько о `Swift Concurrency`. По смыслу это тот же подход: код можно приостановить на ожидании сети, диска или другой async-операции, не блокируя главный поток.

Что особенно важно в `iOS`-приложениях:

- обновление UI должно происходить на главном потоке, поэтому часто используют `@MainActor`;
- сетевые запросы, чтение данных и фоновую работу удобно выносить в `Task`;
- отмена задач важна для экранов, которые могут исчезнуть до завершения загрузки;
- `actor` помогает безопасно работать с разделяемым состоянием без ручной синхронизации.

Пример:

```swift
@MainActor
final class ProfileViewModel: ObservableObject {
    @Published var user: User?

    func load() {
        Task {
            user = try await repository.loadUser()
        }
    }
}
```

В таком коде:

- `Task` запускает асинхронную работу;
- `await` не блокирует UI;
- `@MainActor` гарантирует, что обновление `@Published`-состояния идет безопасно для интерфейса.

## Главное различие между языками

### Кто управляет выполнением

- Python, JavaScript: event loop;
- Kotlin: coroutine scheduler и dispatchers;
- C#: task scheduler и synchronization context;
- Swift: concurrency runtime, `Task` и `MainActor`;
- Rust: executor/runtime;
- Go: встроенный scheduler runtime;
- C++: зависит от библиотеки.

### Блокируется ли поток

Обычно нет, если использовать неблокирующие API. Но если внутри корутины вызвать блокирующую операцию или тяжелый CPU-код, можно все равно заблокировать поток, event loop или executor.

### Stackless vs stackful

Большинство современных моделей `async/await` относятся к stackless coroutines:

- Kotlin;
- Python;
- JavaScript;
- C#;
- Swift;
- Rust;
- C++20.

Go с goroutines ближе к модели легковесных управляемых runtime-потоков.

## Краткий вывод

Почти везде идея одна и та же: корутина позволяет писать конкурентный код как последовательный, сохраняя состояние функции между точками ожидания.

Если упростить:

- Kotlin, C#, Python, JavaScript, Swift, Rust и C++20 в основном используют модель "машина состояний + continuation/future/promise";
- Go использует модель "легковесные runtime-потоки";
- корутины полезны прежде всего для эффективного ожидания, а не автоматически для ускорения CPU-вычислений.
