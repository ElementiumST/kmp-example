# Обзор архитектуры KMP

## Общее описание

Текущая архитектура проекта строится вокруг KMP-ядра с тонкими платформенными оболочками:

- `android/app` — Android Compose-оболочка
- `web-app` — Angular-оболочка
- `ios/app` — SwiftUI-оболочка

Источник истины для активных Gradle-модулей — `settings.gradle.kts`. По нему в проект сейчас входят:

- `kmp/domain`
- `kmp/data`
- `kmp/feature/base`
- `kmp/feature/auth`
- `kmp/feature/contacts`
- `kmp/core`
- `kmp/app`
- `kmp/navigation`
- `kmp/bridge-web`
- `kmp/bridge-apple`
- `kmp/tools/bridge-annotations`
- `kmp/tools/bridge-codegen`
- `kmp/tools/mvi-annotations`
- `kmp/tools/mvi-codegen`

Важно: каталог на диске сам по себе не делает модуль частью сборки. Если директория не подключена в `settings.gradle.kts`, она не является активной частью текущего приложения.

## Текущая структура модулей

### Домен и данные

- `kmp/domain` — доменные модели, контракты репозиториев и use case'ы без привязки к платформе.
- `kmp/data` — реализации доменных контрактов: сеть, слой хранения, хранилище сессии, платформенные клиенты и оркестрация.

Это сильная основа: бизнес-логика отделена от транспорта, БД и UI.

### Рантайм feature-слоя

- `kmp/feature/base` — общий рантайм для feature-слоя.
- `kmp/feature/auth`, `kmp/feature/contacts` — общие feature-компоненты на базе Decompose + MVI.

`feature/base` выступает отдельным слоем общей feature-инфраструктуры. Вокруг него строятся плагины, lifecycle-хуки и общее поведение feature-компонентов.

### Композиция и платформенные модули

- `kmp/core` — корень композиции: запуск, DI, общий root-компонент, корневая навигация и связывание фич.
- `kmp/app` — тонкий входной фасад для платформ, делегирующий в `kmp/core`.
- `kmp/navigation` — отдельная точка подключения навигационных контрактов поверх `kmp/core`.
- `kmp/bridge-web` — bridge-модуль для web: точка входа JS-дистрибутива, TS-определения и сгенерированная таблица маршрутов.
- `kmp/bridge-apple` — Apple-ориентированный фасад и сборка `SharedCore` XCFramework.

На практике `kmp/core` остаётся главным местом общей композиции приложения: здесь сосредоточены запуск, связывание фич и значительная часть bridge-рантайма.

## Направление зависимостей

Текущая зависимость модулей выглядит так:

- `android/app -> kmp/app`
- `android/app -> kmp/navigation`
- `kmp/app -> kmp/core`
- `kmp/navigation -> kmp/core`
- `kmp/bridge-web -> kmp/core`
- `kmp/bridge-apple -> kmp/core`
- `kmp/core -> kmp/data`
- `kmp/core -> kmp/domain`
- `kmp/core -> kmp/feature/base`
- `kmp/core -> kmp/feature/auth`
- `kmp/core -> kmp/feature/contacts`
- `kmp/data -> kmp/domain`

Это направление сохраняет чистые зависимости: бизнес-слой не знает про платформенные оболочки, а платформы не обязаны ходить напрямую в `data` или `domain`.

## Сильные стороны

### 1. Общий код является главным источником поведения

Корневая композиция, навигация, состояние фич и значимая часть бизнес-логики живут в общем коде. Это уменьшает риск расхождения платформ между Android, iOS и Web.

### 2. У проекта понятные границы платформенных адаптеров

У платформ есть явные точки входа:

- Android подключается через `kmp/app` и `kmp/navigation`
- Web собирает и загружает bundle из `kmp/bridge-web`
- iOS потребляет `SharedCore` через `kmp/bridge-apple`

Такое разделение делает платформенную интеграцию более читаемой и предсказуемой.

### 3. Web- и Apple-аспекты bridge-слоя выражены явно

Bridge-логика не торчит из общих feature-интерфейсов. Это важный архитектурный выигрыш: web- и iOS-адаптация остаётся адаптацией, а не просачивается в доменные или feature-контракты.

### 4. Управление lifecycle явно оформлено

В проекте есть явный teardown через `RootComponent.destroy()`, и платформенные оболочки умеют его вызывать. Это важный шаг для Decompose/KMP-приложения, особенно при web- и iOS-интеграции.

### 5. Инструменты вокруг bridge-слоя оформлены системно

`kmp/tools/bridge-annotations`, `kmp/tools/bridge-codegen`, сгенерированная таблица маршрутов и сгенерированные TS-типы формируют стандартизированную схему bridge-интеграции.

## Текущие слабые стороны

### 1. `kmp/core` является центром тяжести

`kmp/core` знает слишком много:

- запуск общего приложения
- корневая навигация
- связывание фич

### 2. `kmp/navigation` пока очень тонкий

С архитектурной точки зрения выделение `kmp/navigation` полезно, но прямо сейчас это скорее пограничный модуль, чем полноценный самостоятельный слой. Если навигация со временем не получит собственные контракты и политику, модуль рискует остаться формальным.

### 3. Web-граница работает через JSON-снимки

Web-интеграция практична, но не полностью типобезопасна на рантайм-границе:

- Kotlin сериализует состояние в JSON
- TypeScript парсит эти строки назад
- UI получает новый снимок целиком

Сгенерированные TS-типы уменьшают риск рассинхронизации на этапе компиляции, но сам рантайм-контракт остаётся строковым.

Сейчас эта проблема находится в разработке
### 4. Рост числа фич может перегрузить слой композиции

Если количество фич заметно вырастет, текущий корень композиции рискует стать тяжёлым:

- больше связывания фич
- больше фабрик и зависимостей
- больше навигационных веток
- больше специфичной для оболочек адаптерной логики рядом с корнем композиции

## От чего архитектура защищает

Сейчас архитектура хорошо защищает проект от типичных проблем роста:

- смена API-клиента или реализации хранения не требует ломать feature/UI-слой
- Android, iOS и Web не обязаны повторно реализовывать feature-flow
- новые фичи можно встраивать через понятные слои: `domain`, `data`, `feature/*`, корень композиции
- мосты для платформ вынесены в более явные пограничные модули

## Практический вывод

Архитектура проекта строится вокруг общего KMP-ядра, отдельных bridge-модулей и тонких платформенных оболочек. Главная зона внимания при дальнейшем развитии проекта — нагрузка на `kmp/core`, потому что именно там сосредоточены композиция приложения, связывание фич и часть bridge-рантайма.

## Дальнейшие цели

- не превращать `kmp/navigation` в формальный alias-модуль, а развивать его как место для реальных навигационных контрактов
- по возможности снижать площадь `api(...)`-экспорта из центральных модулей
- дальше стандартизировать tooling вокруг bridge/codegen/runtime, чтобы меньше логики дублировалось вручную между Kotlin и TypeScript/Swift
